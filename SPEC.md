## なぜこのシステムを作るか

ラップトップ1台とシンプルなフィジカルコントローラーで、ライブハウス・クラブでの音楽と映像のライブパフォーマンスを行うため。

**これは「楽器」であり「音源再現ツール」ではない。**
参考音源は「正解」ではなく、作りたい音のイメージを伝えるための手段として使う。

---

## 絶対に変えない制約

- **リアルタイム性最優先** — 音と映像の遅延は許容しない
- **音源の完全再現はしない** — 近似・雰囲気・方向性を目指す
- **ライブ中の操作はブラウザ GUI のみ** — SC コードの編集は制作・リハーサル時に VS Code + Claude で行う
- **映像ソフトは差し替え可能にする** — SC→OSC→映像ソフトの接合点だけ固定する

---

## コアコンセプト：制御されたカオス

Alva Noto, Autechreのように、**制御されながらも可変性・ランダム性・カオスが共存する**音響空間を目指す。

一般的なDAW（Ableton Live等）の「トラック＋ミキサー＋スコア」という構造主義的アプローチは取らない。代わりに：

- パラメーターが自律的にゆっくり変化し続ける
- 人間はその大きな流れを方向づける
- 2つのモードを切り替えられる：
  - **制御モード** — 人間が細部まで操作する
  - **自律モード** — システムが自律的に進化し続ける

### 決定論的カオス（Deterministic Chaos）

**短期には予測不能、長期には意図した方向へ進む**のが設計の核心。
「完全なランダム」でも「固定したスコア」でもない、第三の状態を目指す。

> あなたは嵐の進路を決める。嵐の形は自然が決める。

これはジャズのコード進行と同じ原理 — コードは固定、即興は自由、調性は守られる。

### 階層的時間制御（Hierarchical Temporal Control）

4つの時間スケールがそれぞれ異なる制御を持つ：

| レイヤー | 時間スケール | 誰が制御するか |
|----------|------------|--------------|
| Song Form | 数分単位 | 人間（Tidal） |
| Section | 8〜32小節 | 人間がプリセット定義 |
| Phrase | 1〜4小節 | 半自動（エンベロープ） |
| Parameter | ミリ秒〜秒 | SCの内部カオス |

上位レイヤーが下位レイヤーの動ける**境界（ceiling/floor）**を定義する。
カオスは境界の中で自由に動くが、境界の外には出られない。

### Bounded Chaosの3つの制御メカニズム

1. **引力点（Attractor）** — カオスが引き寄せられる目標値を設定する。振れ幅は自由だが、中心は常にターゲットに引っ張られる。
2. **カオス幅エンベロープ** — カオスの振れ幅そのものを長期エンベロープで制御する。「静寂 → 緊張 → 解放」という時間的物語を作る。
3. **変異速度（Mutation Rate）** — カオスの変化スピードをTidalが制御する。遅ければ安定したドリフト、速ければグリッチ感。

### 調性制約

音程のカオスはスケール外に出ない。カオスが音程を「自由に選ぶ」のではなく、
**あらかじめ許可された音階の中からランダムに選ぶ**。
短期的には予測不能、長期的には調性が保たれる。

## ライブの構造

- **形式**: シーン切り替え方式
- **時間**: 30〜40分
- **流れ**: あらかじめ複数のシーンを用意し、切り替えのタイミングだけ人間が判断する
- 各シーンはシンセ・エフェクト・ランダム性の度合いのプリセットを持つ

---

## システム全体の流れ（コアパイプライン）

```
[Sonic Anatomy]  ← 楽曲解析・パターン抽出
        ↓
[パターンライブラリ]  リズム・メロディ・コードプログレッション
        ↓ 人間が選択（ブラウザUIのカタログパネル）
[Multi-timescale Algorithm（Python）]
  ├─ 上位（Markov）: 調性・スケール（64拍単位）
  ├─ 中位（Fractal/Markov）: コード度数・和声リズム（小節単位）
  ├─ 下位（L-System）: リズム・旋律装飾（フレーズ単位）
  ├─ メロディ6層制御（スケール整合性保証）
  └─ SCパラメータ制御（音色・テクスチャー）
        ↓
[TidalCycles]  ← 生成されたスコアを演奏
        ↓ OSC
[SuperCollider]  シンセ・サンプラー・エフェクター
        ↓
[Python ブリッジ]
        ↓ WebSocket
[ブラウザ Web UI]
  ├─ パターン選択パネル（カタログ形式）
  └─ SCENEパネル（エネルギー・展開の操作）
        ↓ OSC
[映像ソフト（TouchDesigner / Hydra / 未定）]

[VS Code + Claude]  ← 制作・リハーサル時にコードを編集・改善
```

**Claudeの役割**: 制作・リハーサル時に VS Code チャット経由で SC コード・アルゴリズムを編集・改善する。ライブ中は人間がブラウザ GUI のみで操作する。

## フィードバックループ

人間のフィードバック（「もっと暗く」「リズムが速すぎる」等）を蓄積し、セッションをまたいだ学習に使う。

- ライブ中・制作中のフィードバックはJSON形式で記録
- 蓄積されたフィードバックにより、次回セッションで理想の音により素早く近づけるようにする
- リアルタイムのLLM推論はライブ中の遅延リスクがあるため、学習はセッション間で行う

## 用語定義

| 用語 | 意味 |
|------|------|
| シーン | ライブの一場面。マルチタイムスケールアルゴリズムの振る舞いプロファイルのひとまとまり |
| パターンライブラリ | Sonic Anatomyで解析・抽出したリズム・メロディ・コードプログレッションの集合 |
| Multi-timescale Algorithm | Markov/Fractal/L-Systemを組み合わせた階層的生成アルゴリズム。Tidalスコアと SCパラメータを自律的に進化させる |
| メロディ6層制御 | 調性空間〜表情まで6層で音楽的整合性を保ちながらメロディを生成する仕組み |
| パッチ | SuperCollider上の音響合成の設定一式 |
| grain（グレイン） | グラニュラーサンプラーが扱う音の最小単位（米粒ほどの短い音片） |

## 開発方針

- **TDD（テスト駆動開発）を採用** — ユニット・リグレッション・インテグレーションテストすべてを通過することが「意図通りに動いている」の定義
- **最初に作るもの**: SC ↔ OSC ↔ ブラウザの骨格（細い一本道のパイプ）
- **LLMを最大活用** — 単調なコピペ・繰り返しタスクは極限まで減らす
