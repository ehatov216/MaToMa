## なぜこのシステムを作るか

ラップトップ1台とシンプルなフィジカルコントローラーで、ライブハウス・クラブでの音楽と映像のライブパフォーマンスを行うため。

**これは「楽器」であり「音源再現ツール」ではない。**
参考音源は「正解」ではなく、作りたい音のイメージを伝えるための手段として使う。

---

## 絶対に変えない制約

- **リアルタイム性最優先** — 音と映像の遅延は許容しない
- **音源の完全再現はしない** — 近似・雰囲気・方向性を目指す
- **ライブ中の操作はブラウザ GUI のみ** — SC コードの編集は制作・リハーサル時に VS Code + Claude で行う
- **映像ソフトは差し替え可能にする** — SC→OSC→映像ソフトの接合点だけ固定する

---

## コアコンセプト：制御されたカオス

池田良治・Alva Noto・Autechreのように、**制御されながらも可変性・ランダム性・カオスが共存する**音響空間を目指す。

一般的なDAW（Ableton Live等）の「トラック＋ミキサー＋スコア」という構造主義的アプローチは取らない。代わりに：

- パラメーターが自律的にゆっくり変化し続ける
- 人間はその大きな流れを方向づける
- 2つのモードを切り替えられる：
  - **制御モード** — 人間が細部まで操作する
  - **自律モード** — システムが自律的に進化し続ける

---

## ライブの構造

- **形式**: シーン切り替え方式
- **時間**: 30〜40分
- **流れ**: あらかじめ複数のシーンを用意し、切り替えのタイミングだけ人間が判断する
- 各シーンはシンセ・エフェクト・ランダム性の度合いのプリセットを持つ

---

## システム全体の流れ（コアパイプライン）

```
[VS Code + Claude]  ← 制作・リハーサル時に SC コードを編集
        ↓
[SuperCollider + TidalCycles]
        ↓ OSC
[Python ブリッジ]
        ↓ WebSocket
[ブラウザ Web UI]
        └── GUIパネル（パラメーター操作）
        ↓ OSC
[映像ソフト（TouchDesigner / Hydra / 未定）]
```

**Claudeの役割**: 制作・リハーサル時に VS Code チャット経由で SC コードを編集・改善する。ライブ中は人間がブラウザ GUI のみで操作する。

---

## フィードバックループ

人間のフィードバック（「もっと暗く」「リズムが速すぎる」等）を蓄積し、セッションをまたいだ学習に使う。

- ライブ中・制作中のフィードバックはJSON形式で記録
- 蓄積されたフィードバックにより、次回セッションで理想の音により素早く近づけるようにする
- リアルタイムのLLM推論はライブ中の遅延リスクがあるため、学習はセッション間で行う

## 開発方針

- **TDD（テスト駆動開発）を採用** — ユニット・リグレッション・インテグレーションテストすべてを通過することが「意図通りに動いている」の定義
- **最初に作るもの**: SC ↔ OSC ↔ ブラウザの骨格（細い一本道のパイプ）
- **LLMを最大活用** — 単調なコピペ・繰り返しタスクは極限まで減らす
