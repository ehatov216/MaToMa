## 1. FLUTEとTURINGのパラメーター効きが良くない

**現状の問題:**
- ドローン（持続音）は変化が出にくい音の性格上、パラメーター変化が聞き取りにくい
- CHAOSがfeedback_amtとshimmerにマップされているが、変化が地味
- TURINGの音程変化は8小節に1回でゆっくりすぎる

**改善アイデア:**
- FLUTEのCHAOSをもっと派手なパラメーター（amp変調・ピッチ揺れ）にもマップ
- TURINGのmutation_barsをGUIから調整できるようにする（速さを変えられる）
- パラメーター変化に「ビジュアルフィードバック」を追加してどこが動いているか見えるように

## 6. グリッチ

**目標:** デジタル的な壊れ・欠陥音を音楽的素材として使う

**改善アイデア:**
- **ビットクラッシュ**: `Latch` + 低サンプルレートで量子化ノイズ

- **リングモジュレーション**: 音同士をかけ合わせて金属的な歪み
- **グリッチシーケンス**: ランダムなタイミングで一瞬音が「壊れる」イベント
- GUIに「GLITCH」ボタンを追加 → 確率・強度を制御

## 7. 音楽的調和の基盤（最重要・次回最優先）

**現状の問題:**
現在はドローン・グリッチ・リズムがそれぞれ独立してパラメーターを動かしており、
「ピュアなノイズの集合体」になっている。音楽的な調和・統合性がない。

**SPEC.mdとの対応（階層的時間制御）:**
```
Song Form（数分）  ← Tidal が担う → 未実装
Section（8〜32小節）← シーン切り替え → 実装済み（弱い）
Phrase（1〜4小節） ← 半自動エンベロープ → 未実装
Parameter（ms〜秒）← ChaosEngine → 実装済み
```

**必要な実装（優先順）:**

### 7-1. 共通スケール制約（全音源が同じ音階を共有）
- TURINGが使っているスケール（warm/void/vast/lost のDNA）をドローン・グリッチにも適用
- グリッチ(BURST)のRingz周波数をスケール音程に制約
- 実装: `~currentScale` グローバル変数でスケール共有

### 7-2. コードプログレッション（Phraseレベルの調和）
- 4〜8小節周期で和音の中心が移動する
- TuringGeneを拡張して「コード進行シーケンス」を持たせる
- ドローンのfreqがこの進行に追従する
- 参照: knowledge/rag/sources/generative_music_theory.md

### 7-3. 相対的カオス（大きいカオス）の実装
- 「予定された調和」があって初めて「崩し」が意味を持つ
- FLUTEのTENSION → 調和の崩れ度合い（スケール外に出る確率）
- CHAOSが高いほど和音の境界が曖昧になる
- これがライブ全体の「緊張と解放」を作る

### 7-4. TidalCycles統合（Song Formレベル）
- コード進行・リズムパターンのOSC制御
- bridge.pyのtidal関連コードが未完成 → 完成させる
- 参照: knowledge/rag/sources/tidal_pattern_functions.md

**設計の核心（SPEC.mdより）:**
> あなたは嵐の進路を決める。嵐の形は自然が決める。
> ジャズのコード進行と同じ原理 — コードは固定、即興は自由、調性は守られる。

**優先順位:**
| 優先 | タスク | 効果 |
|------|--------|------|
| 🔴 最高 | 共通スケール制約の実装 | 全音源が調和する |
| 🔴 最高 | コードプログレッション | 音楽的な動きが生まれる |
| 🟡 高 | 相対的カオスパラメーター | ライブの展開を作れる |
| 🟡 高 | TidalCycles統合完成 | Song Formレベルの制御 |

### 8-2. グラニュラーシンセサイザー（アンビエント向け）

**目的:** Abletonグラニュレーターより「途切れない・太い・滑らか」な持続音

**ユーザーの要件:**
- プツプツ途切れない → 高い密度・長いエンベロープ
- サステインがある → ADSR のサステイン重視
- シンセサイザー的な滑らかさ → グレイン間の重なりを十分に確保
- アンビエントに向いている → ドローンとの融合感

**実装アイデア（SCコード参考）:**
```supercollider
// 高密度グラニュラーシンセ（途切れない持続音）
SynthDef(\matoma_gran_synth, {
    arg freq = 220, amp = 0.3, gate = 1,
        grainDur = 0.2,    // 長めのグレイン（重なりを作る）
        density = 40,      // 高密度（途切れを防ぐ）
        spread = 0.3,      // 広がり
        room = 0.6;
    var env = EnvGen.ar(Env.adsr(2.0, 0.5, 0.8, 4.0), gate, doneAction: 2);
    var sig = GrainSin.ar(2, Impulse.ar(density),
        grainDur, freq * (1 + (LFNoise1.kr(0.1) * spread * 0.01)),
        LFNoise1.kr(0.05) * spread);
    sig = FreeVerb2.ar(sig[0], sig[1], mix: room, room: 0.8);
    Out.ar(~effectBus, sig * amp * env);
});
```

## 9. モジュール間相互作用（将来の核心設計）

### 「羊飼い」メタファー（ユーザーの設計哲学）

> 羊（モジュール）同士が相互作用する — 喧嘩したり、仲良くしたり、距離を置いたり。
> 羊飼い（人間・システム）は1つの方向性に向かうことを厳守する。
> 小さなカオス（モジュール間の相互作用）と大きな統合（全体の方向性）。

**SPEC.mdとの対応:**
- 羊同士の相互作用 = Phraseレベルの「半自動エンベロープ」
- 羊飼いの方向性 = Song FormレベルのTidal / シーンの引力点
- Bounded Chaos = 「モジュールは自由に相互作用するが、境界は越えない」

### 具体的な相互作用のアイデア

| 相互作用 | 実装 | 音楽的効果 |
|----------|------|-----------|
| ドローン→グラニュラー | ドローンの明るさ（spectralCentroid）がグレイン密度を制御 | 音の明るさと粒の密度が連動 |
| グリッチ→ドローン | BURSTの発火頻度がドローンのshimmerを一時的に上げる | グリッチが起きるたびにドローンが揺れる |
| リズム→グラニュラー | パーカッションのビートタイミングでグレインのposがジャンプ | リズムとテクスチャが同期 |
| ドローン→グリッチ | ドローンのfeedback_amtが高いときGENDYのchaosが上がる | 音の混濁がグリッチを引き起こす |

### 相互作用の制御設計

```
方向性（羊飼い）: Tidal / シーン引力点 / FLUTEマクロ
    ↓ 境界を設定
相互作用（羊）: モジュール間のパラメーター干渉
    - Organic Couplingの拡張（現在はドローン→グラニュラーのみ）
    - 各モジュールが「受信」できるパラメーターを定義
    - coupling strengthで干渉の強さを制御
    ↓ 境界の中で自由
音響結果: 予測不可能だが統一感のある変化
```

## 10. 【アイデアメモ】羊飼いGUI — 生態系インターフェースの構想

> ⚠️ これはまだアイデア段階。実装の優先度は低いが、MaToMaの長期ビジョンとして記録する。

### 現在のGUI（コックピット型）

- パラメーター・ノブ・スライダーで音楽要素を操作
- DTMプロデューサー・指揮者の視点
- オントロジー：音楽制作の共通言語（BPM・カットオフ・リバーブ等）
- 人間はパラメーターを「設定」する

### 提案するGUI（羊飼い型・生態系型）

コックピットではなく、**生態系の中に直接関与するインターフェース**。

**構成要素の対応：**

| 音楽的要素 | 羊飼い世界の比喩 | GUIでの表現 |
|-----------|---------------|------------|
| 各モジュール（ドローン・リズム・グリッチ） | 羊（個体） | 動く点・生き物のようなオブジェクト |
| モジュール間相互作用 | 羊同士の関係（引力・斥力・群れ） | 線・引力場・距離 |
| コード進行・リズム方向性 | 羊飼いが指す方向 | 矢印・引力点・地形の傾き |
| エフェクト・モジュレーション | 環境（雨・風・雷） | 背景の変化・力場・粒子 |

**モジュール間相互作用のパターン（関数で表現）：**
- 戦型的（競合）: 2つのモジュールが互いに周波数を避ける
- 非戦型的（共鳴）: 倍音関係にある音程に引き寄せられる
- 群れ行動: 複数モジュールが同じ方向に少しずつ動く
- 捕食関係: あるモジュールが別のモジュールを「吸収」して変形する

**インターフェースのイメージ：**
- 画面上に羊（モジュール）が動いている
- 人間はその世界の「傾き」（引力点・方向性）を設定する
- 細かいパラメーターを触るのではなく、世界の物理法則を変える
- 「雷を落とす」= グリッチを強化する
- 「草原の方向を変える」= コード進行を切り替える

### 現在のGUIとの共存案

- 現在のコックピットGUI：細かい制御が必要なリハーサル・制作時
- 羊飼いGUI：ライブパフォーマンス時の直感的操作

### 技術的な実現可能性メモ

- フロントエンド: Canvas APIまたはThree.jsでビジュアル
- 物理演算: 引力・斥力・群れ行動はシンプルな粒子シミュレーション
- OSCへの変換: 羊の座標・距離・速度 → パラメーター値
- 既存のWebSocket + bridge.pyの構造はそのまま使える
