# MaToMa 段階的開発ロードマップ

> 最終更新: 2026-04-12
> SPEC.mdの「制御されたカオス」と「階層的時間制御」を土台に、最小構成から段階的に積み上げる。

---

## 全体方針

**増築方式**: 各Phaseは前のPhaseの上に乗る。既存の動作を壊さずに機能を追加する。
**検証優先**: 各Phaseは「聴いて確認できる」状態で完了する。
**音楽的整合性**: どのPhaseでも、出力される音はスケール・調性を守る。

---

## Phase 1 — 最小演奏骨格（MIDIキーボード的）

**ゴール**: SA解析したメロディーパターン（音程列）とリズムパターンを、  
単一のSCシンセに流して演奏できる状態。

### 何をするか

```
Sonic Anatomy DB
  → 音程列（例: [C4, E4, G4, A4, ...]）
  → リズムパターン（onset_density, BPM相当）
       ↓ TidalCycles の note パターンに変換
  → d1 に単一SynthDef を割り当て
       ↓ SCに流れる
  → 音が出る
```

### 実装内容

- [ ] SA解析結果の音程列をTidalの `note` パターンとして変換するロジック確認（`sonic_anatomy_bridge.py`）
- [ ] TidalCycles の `d1` に単一のSCシンセ（`\matoma_basic` または専用Lead SynthDef）を対応付け
- [ ] リズムパターン（euclid, speed）をTidalパラメーターにマッピング確認
- [ ] UIから「SAを適用して再生」→「音が出る」の1本道を通す

### 完了条件

- SAカタログから曲を選び「適用」すると、そのメロディーとリズムが単一シンセから聞こえる
- 音程はスケール内（C major, A minor等）に収まっている
- 停止・再生が確実に動作する

### 現在の実装との対応

- `MusicGenerator._lower_tick()` がTidalにコードを送る仕組みは存在する
- `SonicAnatomyBridge` がSA DBから音程・リズムを抽出する仕組みは存在する
- **確認すること**: 音程列が正しくTidal `note` パターンになっているか、単一シンセが正しく鳴っているか

## Phase 2 — マルチパート分割

**ゴール**: 同じキー・スケール・コードトーンの中で、複数の「役割」に分かれた演奏。

### 4つのパート役割

| パート | 役割 | 対応するSCシンセ | Tidalチャンネル |
|--------|------|----------------|---------------|
| Lead   | メロディーの主旋律（高音域・動きが多い） | サブトラクティブ or FM | d1 |
| Pad    | 和音の持続音（動きが少ない・広がり重視） | ドローン or グラニュラー | d2 |
| Pluck  | 短い発音・装飾音（アタック強め） | 物理モデリング or グラニュラー | d3 |
| Bass   | 低音域の根音（シンプル・リズム強調） | サブトラクティブ（低域） | d4 |

### 実装内容

- [ ] 各パート用SCシンセ（SynthDef）の定義または既存シンセの役割割り当て
- [ ] `MusicGenerator` に「パート分配」ロジックを追加
  - 同じスケールの中で各パートに異なる音域・パターンを割り当てる
  - Lead: 高音域・速いパターン / Bass: 低音域・遅いパターン
- [ ] TidalCycles の d1〜d4 をそれぞれのパートSynthDefにルーティング
- [ ] UIに「各パートのON/OFF」スイッチを追加

### 完了条件

- 4パートが同じキー・スケール内で独立したメロディーを演奏している
- 各パートを個別にON/OFFできる
- 全パートが同時に鳴っても音楽的な整合性がある（濁らない）

## Phase 3 — 自律的メロディー生成

**ゴール**: SAのメロディーをベースに、マルコフ連鎖・Lシステムで  
「ゆっくりと変化するメロディー」を生成する。音楽理論の制約の中で動く。

### アーキテクチャ

```
SA解析メロディー（固定起点）
  ↓
Markov連鎖（コード進行の遷移）
  → 上位: スケール・調性の緩やかな移動（64拍単位）
  → 中位: コード度数の選択（小節単位）
  ↓
L-System（フレーズの変形・装飾）
  → リズム・旋律の微細な変化（フレーズ単位）
  ↓
各パートへ送信
```

### 音楽的制約（ゲシュタルト崩壊防止）

- スケール外の音程に出ない（厳守）
- 1ステップでの調性変化は半音2つ以内
- Markov遷移は自然な音楽進行テーブル（`DEGREE_TRANSITIONS`）に従う
- Lシステムの変異確率は低め（0.1〜0.3）
- 「予定された調和」（コード進行のグリッド）が常にある

### 実装内容

- [ ] `MusicGenerator` の既存Markov実装の動作確認・整理
- [ ] L-System フレーズ変形の実装（既存 `_lower_tick` を拡張）
- [ ] スケール整合性チェック関数（音程がスケール内か検証）
- [ ] ユーザーUIに「コード選択パネル」を追加
  - 現在のキー・スケールに基づいたコードの選択肢を提示
  - ユーザーが選んだコードにMarkovの引力点をシフト
  - 選択は「方向の指示」であり、即時切り替えではない

### 完了条件

- 放置していると音楽がゆっくり変化していく（5〜10分スケール）
- 変化の前後で音楽的整合性が保たれる（突然外れない）
- UIのコード選択が音楽の展開に緩やかに影響する

## Phase 4 — エフェクターレイヤー

**ゴール**: 各パートに適切なエフェクトを配置し、音空間を整える。  
エフェクターパラメーターもMarkov・Lシステムで緩やかに変化する。

### パート別エフェクト哲学

| パート | エフェクト哲学 | 使うUGen |
|--------|-------------|---------|
| Lead   | 空間（音の位置） | Reverb（GVerb, short）+ Delay |
| Pad    | 時間（音の持続） | GVerb（long reverb）+ Chorus |
| Pluck  | 輪郭（音の形） | Delay（タップ系）+ 軽いReverb |
| Bass   | 圧（音の重さ） | Compressor + SubHarmonic |

### バス設計

```
d1 Lead  → FX Bus A (short reverb, delay)
d2 Pad   → FX Bus B (long reverb)
d3 Pluck → FX Bus C (delay tap)
d4 Bass  → FX Bus D (dry + sub)
  → Master Bus → Limiter → Out
```

### 実装内容

- [ ] SCのバス設計（FX Bus A〜D）を `sc/effects.scd` に追加
- [ ] 各パートシンセをFXバスにルーティング
- [ ] `ThreeLayerController` のエフェクターパラメーター制御の追加
  - リバーブのroom/revtime をMarkovで制御
  - ディレイタイムをLシステムで制御（拍の分数として）
- [ ] UIに「エフェクト深度」スライダーを追加（wet/dry比）

### 完了条件

- 各パートが適切な空間感を持つ
- エフェクターパラメーターが自然にゆっくり変化する
- マスターバスでクリッピングが起きない

## Phase 5 — シンセパラメーター自律変調

**ゴール**: マルコフ・Lシステムがシンセサイザーのパラメーターも  
ゆっくりと変化させる。複数レイヤーが有機的に絡み合う。

### 変調の対象（ゲシュタルト崩壊防止のため範囲を限定）

| パラメーター | 変化のスケール | 変化の単位 |
|------------|-------------|---------|
| Lead.cutoff | 500〜3000Hz | 小節単位でドリフト |
| Pad.revtime | 8〜20秒 | セクション単位 |
| Pluck.grainDur | 0.05〜0.3秒 | フレーズ単位 |
| Bass.amp | 0.3〜0.8 | 小節単位 |

### 制約（不変）

- 音程・スケール制約は変わらない
- 一度に変化するパラメーター数は全体の1/3以下
- 変化量の上限（ceiling/floor）はシーン定義に従う
- クリッピング・ノイズ爆発は発生しない

### 実装内容

- [ ] `ThreeLayerController` のSCパラメーター制御を各パートに拡張
- [ ] `PARAM_SPECS` に各シンセパラメーターを追加
- [ ] パラメーター変化をOSC経由でSCに送信するルーティング確認
- [ ] 「音楽的変化」と「パラメーター変化」の時間スケール分離
  - メロディー変化（Markov）: 64拍単位
  - パラメーター変化（BoundedWalk）: 小節単位
  - 微細変動（Dejavu）: フレーズ単位

### 完了条件

- 30分放置しても音楽が崩壊しない
- パラメーターの変化が音楽的な文脈に沿って感じられる
- ライブパフォーマンスとして最初から最後まで通せる

## Phase間の移行基準

各Phaseの「完了条件」を満たしたと判断したら次に進む。  
移行前に必ず「30分通し演奏」で破綻がないことを確認する。

```
Phase 1 → 2: 単一シンセで30分聴ける音楽が出せる
Phase 2 → 3: 4パート同時演奏が30分安定する
Phase 3 → 4: メロディー変化が音楽的に自然（突然外れない）
Phase 4 → 5: エフェクト込みで30分安定する
Phase 5 完了: ライブで使える状態
```
