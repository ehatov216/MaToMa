## 1. 設計哲学：羊飼いモデル（更新版）

### なぜこの設計か

人間が「何の音を鳴らすか」を逐一決めるのではなく、**音楽が自律的に生成・進化し続ける状態を作り、人間はその大きな方向だけを操作する**。

### 羊飼いの役割（新）

| 道具 | 操作 | 音楽的意味 |
|------|------|-----------|
| パターン選択（カタログUI） | Sonic Anatomyから起点を選ぶ | 「どのフィールドで羊を放すか」を決める |
| SCENEボタン | シーン（感情的世界観）を切り替える | 「群れの向かう方向」を大きく変える |

### 設計判断の基準

新機能を追加するとき、常にこの問いで判断する：
> 「これは羊飼いの道具か？それとも羊を直接操る鎖か？」

音楽的な細部（どの音程か、どのタイミングか）はアルゴリズムが決める。人間はそれを起動・方向づけるだけ。

## 3. Multi-timescale Algorithm の3層構造

### 3.1 上位：マルコフ連鎖（Markov）— 調性・スケール決定

- **時間軸**: 64拍（≒ 8小節 at 120BPM）〜楽曲全体
- **役割**: キー、スケール、モードの決定
- **入力**: Sonic Anatomyで抽出したコードプログレッション
- **出力**: 次の「調性世界（Scale）」の予約

```python
# 遷移確率表（例）: C major → 次のスケール
transition_matrix = {
    "C_major": {"C_major": 0.4, "A_minor": 0.3, "F_major": 0.2, "G_major": 0.1},
    "A_minor": {"A_minor": 0.4, "C_major": 0.3, "D_minor": 0.2, "E_minor": 0.1},
    ...
}
```

### 3.2 中位：フラクタル/マルコフ（Fractal/Markov）— コード度数・和声リズム

- **時間軸**: 1〜4小節（10〜30秒）
- **役割**: スケール内のどの和音を使うか（I/IV/V等）、和音の切り替え速度
- **入力**: 上位のScaleと、Sonic Anatomyのコードプログレッション起点
- **制約**: Gravity Matrixで調性引力を保証（後述）

### 3.3 下位：Lシステム（L-System）— リズム・旋律装飾

- **時間軸**: フレーズ単位（1〜2小節）
- **役割**: 中位が選んだコードトーンを核に、音の長さ・休符・装飾音を展開
- **入力**: 中位のコード、Sonic Anatomyのリズムパターン起点
- **制約**: Dynamic Clampingで音域制御（後述）

## 4. メロディ6層制御ヒエラルキー

リズムはマルチタイムスケールアルゴリズムで比較的自由に制御できるが、メロディは制御なしでは調性破綻のリスクがある。このため、メロディ専用の6層制御を導入する。

### 層の定義

| 層 | 名称 | 時間軸 | 支配内容 | 本質 |
|----|------|--------|----------|------|
| 1 | 調性空間（Tonal Context） | 楽曲全体〜セクション（数分〜30秒） | キー・スケール・モード | 「帰るべき場所」と「使える音の色彩」の定義 |
| 2 | 和声リズム（Harmonic Rhythm） | セクション〜4小節（10〜20秒） | 和音が切り替わる速度とタイミング | エネルギーの更新頻度。予測しやすさの支配 |
| 3 | 和音機能と度数（Harmonic Function） | 1小節〜1拍（1〜数秒） | スケール内の和音選択（I/IV/V等） | 緊張と緩和。「安心」か「期待」かの決定 |
| 4 | リズム・モチーフ（Rhythmic Pattern） | 1〜2小節 | 音の長さ・休符・シンコペーション | メロディの「骨格と運動性」 |
| 5 | 旋律輪郭（Melodic Contour） | 1〜2小節 | 音域・跳躍進行か順次進行か | 音の「物理的シルエット」 |
| 6 | 表情と質感（Articulation & Sound Design） | 瞬間〜音符単位（ms） | ベロシティ・エフェクト・Pan | 音符に「生命」を吹き込む |

### 制御の方向性（双方向）

上位層が下位層を制約しながら、下位層の連続が上位層を「裏切る」こともある：

```
トップダウン（定石）:
  Cメジャーキー（層1） → ド（層3）で終わる → 安心感（層6）

ボトムアップ（裏切り）:
  層6で強烈な歪みやピッチベンド
  → 本来層1が決めていた安定をあえて壊す
  → 聴衆は新しいコード感（層3）を想起する
```

## 5. 破綻回避の4機構

### 5.1 遅延評価（Lazy Evaluation）— スケール不一致の防止

上位が命令を出し下位が即座に受けるのではなく、**「下位が次のサイクルに入る瞬間に上位の最新状態をサンプリングする」**プル型の同期。

```
グローバル・クロック:
  Tidalの cycle 境界でのみ上位層（Markov）の遷移を許可

「先行書き込み」と「確定」の分離:
  上位Markovは次の遷移先を「予約」だけ保持
  中位のコード選択がI度（またはcycle先頭）に到達した瞬間に「確定」
  → 音楽的文脈を壊さずに調性を切り替えられる
```

```python
class GlobalClock:
    pending_scale: Optional[Scale] = None   # 予約中
    current_scale: Scale                    # 確定済み

    def on_cycle_start(self):
        if self.pending_scale is not None:
            self.current_scale = self.pending_scale
            self.pending_scale = None
```

### 5.2 グラビティ・マトリクス（Gravity Matrix）— 調性感の保持

単一の遷移確率表ではなく、**現在の和音度数に応じた動的な重み付けマトリクス**。

```
機能的マルコフ連鎖:
  「ドミナント状態」用テーブルと「トニック状態」用テーブルを持つ

  V度の状態にいる時: I度への遷移確率を 0.8 に跳ね上げ
  → 数学的ランダム性の中に「音楽的引力」が生まれる
  → アルゴリズム特有の「浮遊感すぎる問題」を解消
```

```python
gravity_matrix = {
    "dominant": {"I": 0.80, "IV": 0.10, "VI": 0.05, "other": 0.05},
    "tonic":    {"I": 0.20, "IV": 0.35, "V": 0.30,  "VI": 0.15},
    "subdominant": {"I": 0.15, "V": 0.50, "IV": 0.20, "other": 0.15},
}
```

### 5.3 音域の動的クランプ（Dynamic Clamping）— 音域逸脱の制御

音域逸脱をエラーとして除去するのではなく、**Energyパラメータと連動した「音楽的緊張感」として利用**。

```
Energy 低: 下位ボイシングの可動域を1オクターブ以内に制限（安定感）
Energy 高: 下位ボイシングの可動域を3オクターブまで解放（カオス・高揚）

音域外に音が配置されそうな場合:
  → エラーにせず「最も近いスケール内の音かつ許容レンジ内」に折り返す（Wrap-around）
  → 破綻が「意図的な跳躍」に変換される
```

```python
def clamp_pitch(pitch: int, scale: Scale, energy: float) -> int:
    octave_range = 1 + energy * 2  # 1〜3オクターブ
    center = scale.root_octave
    min_pitch = center - octave_range * 6
    max_pitch = center + octave_range * 6
    # 範囲外なら最近傍のスケール音に折り返す
    return scale.nearest_note(clamp(pitch, min_pitch, max_pitch))
```

### 5.4 スケール・クォンタイザー（Scale Quantizer）— 最終安全網

層6の後、最終出力の直前で**現在のスケール（層1）で強制的に再クォンタイズ**する。

```
どんなにアルゴリズムで同期させても、遷移の瞬間にはmsラグで外れ値が出る可能性がある。
これを最後の一線で物理的にフィルタリングする = 聴感上の「事故」を100%防ぐ「ガードレール」。
```

### 統合制御フロー

```
上位（Markov）:     64拍ごとに「次のScale」を予約
     ↓ Lazy Eval
同期点（Cycle先頭）: 予約をcurrent_scaleに確定
     ↓ Gravity Matrix
中位（Fractal/Markov）: Scaleに適合した「引力付き度数」を選択
     ↓ Dynamic Clamping
下位（L-System）:   コードトーンを核に、Energyに応じた音域で装飾を展開
     ↓ Scale Quantizer
最終出力:           全ノートを現在のScaleで再検閲・微調整
```

## 6. Sonic Anatomyとの接続

### 役割

Sonic Anatomyで解析した実際の楽曲パターンが、マルチタイムスケールアルゴリズムの**起点リファレンス**になる。

```
Sonic Anatomy が提供するもの:
  - リズムパターン（ドラム・パーカッション）
  - メロディパターン（音程の輪郭・リズムモチーフ）
  - コードプログレッション（度数・機能進行）

Multi-timescale Algorithm がそれを受けて:
  - パターンを「起点」としてMarkovチェーンの初期状態を設定
  - アルゴリズムが徐々にそこから変異・進化させていく
  - 「元の曲に似ているが同じではない」音楽が生成される
```

### 人間の操作（パターン選択UI）

ライブ前/ライブ中に、カタログUIから起点パターンを選ぶ：

```
[リズムパターン]   [メロディパターン]   [コードプログレッション]
  pattern_01         motif_03            prog_Cm
  pattern_02         motif_07            prog_Am
  pattern_05         motif_12            prog_Gmaj
  ...                ...                 ...
```

選択後、アルゴリズムが起動してTidalスコアを自動生成・進化させる。

## 7. SuperColliderパラメータの制御

SCパラメータ（音色・テクスチャー）も、メロディと同じマルチタイムスケールアルゴリズムで制御する。

### 時間スケールとパラメータの対応

| 時間スケール | アルゴリズム | 制御するSCパラメータ |
|------------|------------|-------------------|
| 数分〜全体 | 上位Markov | シーンの音色プロファイル（reverb room / 音域重心） |
| 小節〜セクション | 中位Fractal | cutoff / feedback_amt / grain density |
| フレーズ〜瞬間 | 下位L-System / LFNoise | pitch detune / pan / grain spray |

旧設計の Layer A（LFNoise呼吸・心拍・神経）と Layer B（Turing Gene）は、この枠組みに吸収される。LFNoiseは「下位L-System」の時間スケール実装として残る。

### 安全制御（変更なし）

音響的な安全制御は旧設計から継承する：

```
[Feedback パス]
  └─ LeakDC.ar → DCオフセット蓄積を防ぐ
  └─ Clip.ar(0.0, 0.72) → 発振しない上限を設ける

[SynthDef 最終出力]
  └─ Limiter.ar(sig, 0.9, 0.01) → 何があっても 0.9 超えない
```
